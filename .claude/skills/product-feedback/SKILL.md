# Product Feedback Analysis

Анализирует обратную связь пользователей FitFlow на основе данных из таблицы `feedback` (SQLite, `data/fitflow.db`). Выявляет боли, тренды NPS, сегментирует по аудиториям и формирует рекомендации.

## Когда использовать

```
Запрос пользователя?
│
├─ "анализ проблем FitFlow"         → Полный цикл
├─ "анализ фидбека"                 → Полный цикл
├─ "что болит у пользователей"      → Полный цикл
├─ "отзывы пользователей"           → Полный цикл
├─ "/product-feedback"              → Полный цикл
├─ "NPS FitFlow"                    → Полный цикл
└─ "feedback analysis"              → Полный цикл
```

## Аргументы

- **days** (опционально, по умолчанию `60`): период анализа в днях

## Инструменты

- **Python-скрипт:** `python3 .claude/skills/product-feedback/product_feedback_analysis.py` — пакетное выполнение всех SQL-запросов к SQLite
- **База данных:** `data/fitflow.db` — таблица `feedback` с отзывами пользователей (NPS-опросы, оценки магазинов, in-app фидбек)
- **Документация данных:** `data/README.md` (секция про таблицу `feedback`)

## Процесс анализа (token-optimized)

Процесс разделён на 2 фазы: compact stats в основной контекст + subagent для текстовых отзывов.

### Шаг 1. Запуск анализа

```bash
python3 .claude/skills/product-feedback/product_feedback_analysis.py --db data/fitflow.db --days {{days}} --output /tmp/feedback_fitflow.json
```

**Что происходит:**
- **stdout** (попадает в основной контекст): compact JSON — только количественные данные (summary, weekly_trend, mark_distribution, nps_by_channel, nps_by_persona). Без текстовых отзывов.
- **`/tmp/feedback_fitflow.json`** (файл): полный JSON со всеми текстовыми отзывами. Читается subagent'ом.

| В stdout (compact) | В файл (full) |
|---------------------|---------------|
| `summary` — статистика | Всё из stdout + |
| `weekly_trend` — тренд NPS | `negative_feedback` — тексты mark 1-2 |
| `mark_distribution` — оценки 1-5 | `neutral_feedback` — тексты mark 3 |
| `nps_by_channel` — NPS по каналам | `positive_feedback` — тексты mark 4-5 |
| `nps_by_persona` — NPS по персонам | |

### Шаг 2. Категоризация отзывов (subagent)

Запусти **Task(subagent_type=general-purpose, model=haiku)** для анализа текстовых отзывов:

```
Промпт для subagent:

Прочитай файл /tmp/feedback_fitflow.json и проанализируй текстовые отзывы
из секций negative_feedback, neutral_feedback и positive_feedback.

**Задача 1 — Категории болей** (из negative_feedback + neutral_feedback):
Категоризируй по темам. Для каждой категории верни:
- Название категории
- Количество упоминаний (примерное)
- 2-3 характерные цитаты (дословно из message)
- Severity: Critical / High / Medium / Low
- Компонент продукта (если определяется)

Типичные категории для фитнес-приложения: Производительность, Контент тренировок,
Интерфейс, Подписка/цена, Баги, Мотивация/геймификация, Персонализация.
Адаптируй под конкретный продукт.

**Задача 2 — Что хвалят** (из positive_feedback):
Выдели 3-6 ключевых тем с 1-2 цитатами на каждую.

Верни результат в markdown (секции 5 и 6 шаблона отчёта).
Не добавляй ничего от себя — только то, что есть в отзывах.
```

### Шаг 3. Сборка отчёта

Объедини:
- Compact stats из stdout (шаг 1) → секции 1-4
- Результат subagent (шаг 2) → секции 5-6
- Рекомендации на основе обоих → секция 7

На основе полученных данных сформируй:

**3.1. Executive Summary:**
- Общий NPS-like score и средняя оценка
- Тренд: растёт / падает / стабилен (на основе weekly_trend)
- Объём выборки: кол-во уникальных пользователей, кол-во оценок
- Распределение по каналам (nps_survey, in_app_feedback, google_play_review, app_store_review, support_ticket)

**3.2. Сегментный анализ:**
- Какие каналы дают больше всего оценок? Где NPS ниже среднего?
- Какие persona — зона риска (NPS < 25%)?
- Крупнейший сегмент vs наименее довольный

### Шаг 4. Формирование отчёта

Выведи структурированный отчёт в markdown:

```
## Анализ обратной связи: FitFlow (за последние {{days}} дней)

**Дата:** {{YYYY-MM-DD}}
**Источник:** data/fitflow.db (таблица feedback)
**Период:** последние {{days}} дней ({{start_date}} — {{end_date}})
**Формула NPS:** `100 × (mark=5 − mark≤2) / total`

### 1. Executive Summary

### 2. Тренд NPS по неделям

### 3. Распределение оценок

### 4. NPS по сегментам
(каналы + persona)

### 5. Категории болей
(из subagent)

### 6. Что хвалят
(из subagent)

### 7. Рекомендации
(синтез обоих источников)
```

### Шаг 5. Сохранение

Запиши в `researches/feedback-analysis-{{YYYY-MM}}/REPORT.md`.

## Формула NPS-like score

Адаптированная формула для 5-балльной шкалы:

```
NPS = 100 * (COUNT(mark = 5) - COUNT(mark <= 2)) / COUNT(mark)
```

Шкала:
- **Promoters:** mark = 5 (только высший балл)
- **Passives:** mark IN (3, 4) — не участвуют в расчёте, но входят в total
- **Detractors:** mark IN (1, 2)

## Anti-patterns

- **Не анализировать только средний балл** — NPS информативнее (учитывает распределение)
- **Не игнорировать объём выборки** — NPS при <30 оценках ненадёжен, указывать `[⚠️ Малая выборка]`
- **Не читать только негатив** — позитивный фидбек показывает сильные стороны, которые нельзя ломать
- **Не считать каждый отзыв отдельной проблемой** — группируй по темам, считай упоминания
- **Не смешивать каналы** — отзывы в App Store могут отличаться от NPS-опросов по тональности
- **Не выдавать рекомендации без привязки к данным** — каждая рекомендация опирается на категорию боли с цитатами
